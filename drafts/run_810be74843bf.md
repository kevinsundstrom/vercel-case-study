# Durable execution for long-running agents

Long-running agents fail at the boundaries: a network timeout mid-call, a process restart between steps, a server crash after an LLM response arrives but before it gets written anywhere. The result is an agent that restarts without knowing what it already did. Both Vercel Workflows and Cloudflare Workflows solve this, but their approaches differ at the architectural level, not the API surface.

## The mechanics of recovery

Both platforms implement replay-with-memoization, a pattern originating from Temporal, an open-source durable execution engine. When a workflow resumes after failure, the runtime re-executes the orchestrating function from the top. Completed steps return cached results without re-running. Incomplete steps execute normally.

## Where durability lives

Vercel implements durability through two TypeScript directives applied to async functions. The first marks the orchestrating function; the second marks each individual step.

```typescript
import { task } from "@vercel/workflow";

export const myWorkflow = task(async (event) => {
  const user = await task(async () => {
    return await fetchUser(event.userId);
  });

  const response = await task(async () => {
    return await callLLM(user.data);
  });

  return response;
});
```

The runtime intercepts step calls, records their results in a persistent log, and on resumption returns cached results without re-executing. Step boundaries are declared in code structure rather than configuration.

Cloudflare implements durability through Durable Objects, a primitive that pairs state, compute, and co-location in a single instance. Each workflow instance is one such object, and steps are identified by explicit string names.

```typescript
export class MyWorkflow extends WorkflowEntrypoint {
  async run(event: WorkflowEvent, step: WorkflowStep) {
    const user = await step.do("fetchUser", () => fetchUser(event.payload.userId));
    const response = await step.do("callLLM", () => callLLM(user));
    return response;
  }
}
```

The `step.do()` method checks for a cached result by name, returns it if present, or executes, stores, and returns the result if not. Cloudflare's engineers have described this as a game loop: the function re-runs until every named step has a cached outcome. Step identity here is the string you provide, not the code location.

## Where state lives

Vercel stores workflow state in a managed event log. The storage layer is swappable through an adapter system called Worlds. Because state is serialized to an event log, a client can disconnect from a running workflow and reconnect later without losing progress. The SDK is open source, and the runtime is portable across deployment targets.

Cloudflare stores workflow state in a SQLite database inside each Durable Object instance. State and compute are co-located at the edge. For agents that accumulate state across many steps, that co-location reduces network round-trip latency for each read and write. There is no self-hosted path. Cloudflare Workflows runs on Cloudflare.

## Step identity, scaling, and determinism

Vercel generates step IDs from filepath and function name. Renaming or moving a function generates new IDs in the next deployment. In-flight runs finish on the deployment they started on, a mechanism Vercel calls deploy skew protection. A renamed step will execute again on resumption rather than returning a cached result.

Cloudflare requires explicit string names for steps. The string is the identity. Two different operations with the same string will collide silently. The compiler will not catch it.

Cloudflare's co-located state model reduces latency for workflows that read and write accumulated state frequently. Vercel's event-log model carries overhead as histories grow. Vercel's documentation recommends splitting into child workflows once an event log exceeds approximately 2,000 entries.

Both platforms require orchestrating functions to be deterministic. No `Math.random()`, no `Date.now()`, no side effects outside of explicitly marked steps, because the replay mechanism re-executes the orchestrating function from the top and depends on identical outcomes across runs.

## Visibility and portability

Vercel makes orchestration invisible to your application structure. Directives extend existing async functions, so a function with step calls remains a standard async function with no separate entrypoint, dedicated class, or configuration file required.

Cloudflare makes orchestration explicit. A workflow is a dedicated class with its own entrypoint and lifecycle, existing as a distinct architectural layer separate from the rest of your application code.

On portability: Vercel's open-source SDK and pluggable storage mean the workflow layer can outlive the deployment layer. The storage adapter is the swap point. Cloudflare binds workflow state, compute, and runtime together into a single platform primitive. Teams already building on Cloudflare Workers get low-latency co-located state as a direct consequence of that model. If you are evaluating platforms from the outside, it is the first thing to pressure-test before you commit.

## The architectural decision

The network timeout, the process restart, the crash after the LLM responds but before anything is written. Both platforms handle those failure modes. Vercel handles them through directives that annotate existing async functions, with state in a portable event log, and step identity tied to code location. Cloudflare handles them through a dedicated class and co-located SQLite state, with step identity tied to the string you name explicitly. Transparent orchestration with a portable runtime and explicit orchestration with co-located state are different bets, and which one fits depends on what your agent needs to do and where it needs to run.
