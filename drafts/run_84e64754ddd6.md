# Vector databases for production RAG: Pinecone vs. Weaviate vs. Qdrant vs. pgvector

Most comparison posts hand you benchmark tables and call it done. This article focuses on operational questions: what breaks at scale, what the pricing model punishes, and which constraints are architectural versus fixable in a future release.

---

## 1. The four options and where they sit

### Pinecone

Pinecone is a proprietary managed vector database with two distinct tiers. **Serverless** launched in 2023 and bills per request — no capacity planning required. **Pod-based** requires upfront allocation across three pod types: s1 (CPU-optimized), p1 (GPU-accelerated), and p2 (next-generation GPU). The two tiers share an API but have different performance and cost profiles. Vector storage is physically separate from metadata storage, a fact that shapes filtering behavior throughout the stack.

### Weaviate

Weaviate is Apache 2.0-licensed and self-hostable on Docker or Kubernetes. The managed offering, **Weaviate Cloud** [FACT-CHECK: Research notes refer to this as "Weaviate Cloud Service (WCS)" — the article's shortening to "Weaviate Cloud" may be inaccurate if the official product name differs], handles cluster operations and multi-region deployment. Its built-in vectorizer plugins connect directly to OpenAI, Cohere, and HuggingFace. You can push raw text and receive stored vectors without a separate embedding step. The native API is GraphQL, with REST available as an alternative.

### Qdrant

Qdrant is open-source (AGPL with a commercial license option) and written in Rust. The managed offering is **Qdrant Cloud**. The Rust implementation produces measurably lower per-query overhead than equivalent Go or Python vector stores. Qdrant supports gRPC in addition to REST, which matters for high-throughput scenarios. Its payload (metadata) filtering engine handles complex boolean queries during search traversal rather than as a post-processing step.

### pgvector

pgvector is a PostgreSQL extension, not a standalone database. You can run it on any Postgres host: Neon, Supabase, AWS RDS, Azure Database for PostgreSQL, Railway, or self-hosted. Vectors are stored in ordinary table columns alongside relational data. All operations are SQL. ACID guarantees apply to vectors and relational data uniformly. There is no separate vector storage tier and no additional licensing cost.

## 2. Query latency and throughput

### Benchmark caveats

Published benchmark numbers depend heavily on index size, vector dimensionality, hardware, filter selectivity, concurrent query volume, and network path. Vendor-published numbers reflect best-case configurations. ANN-Benchmarks (github.com/erikbern/ann-benchmarks) provides community-maintained results on standard datasets (SIFT, GIST), though coverage of managed cloud tiers is inconsistent. Use the figures below as relative indicators, not absolute targets.

### p50 / p99 latency across index sizes

| Configuration | p50 latency | p99 latency | Notes |
|---|---|---|---|
| Pinecone Serverless | 100–150 ms | 300–500 ms | Cold starts add 1–5 s after idle |
| Pinecone Pod (p1/p2) | 50–100 ms | 150–300 ms | Warm memory, no cold start |
| Weaviate Cloud | 80–150 ms | 300–600 ms | Stable if index fits in memory |
| Qdrant Cloud | 50–100 ms | 150–300 ms | gRPC path reduces overhead further |
| pgvector on Neon | 150–250 ms | 500–1500 ms | Neon cold starts dominate p99 |
| pgvector on RDS | 50–100 ms | 200–400 ms | Comparable to dedicated stores |

Latency at 1M vectors is roughly similar across all options. Divergence appears at 10M and becomes significant at 100M. Qdrant maintains stable latency through 100M+ vectors when collections are properly sharded. Pinecone Serverless shows the steepest degradation at the high end.

### Effect of index type

HNSW is the primary index type for Pinecone, Weaviate, and Qdrant. pgvector supports both HNSW and IVFFlat. Key trade-offs:

- **HNSW**: Low query latency, high memory usage, no retraining when you add vectors.
- **IVFFlat** (pgvector): Lower memory, but you must choose the `lists` parameter at index creation. Accuracy drops if `lists` is mistuned relative to dataset size.
- **HNSW + scalar quantization** (Qdrant): Reduces memory 4–10× with a small accuracy trade-off. Query latency drops 1.5–2× compared to unquantized HNSW. This is Qdrant's strongest operational advantage for large indexes.

### Filtering behavior

Pre-filter applies metadata constraints before ANN search, reducing the candidate set. Post-filter applies them after, then discards non-matching results. Pre-filter is faster when the filter is highly selective. Post-filter degrades when few vectors survive.

| Database | Filtering approach | Predictability |
|---|---|---|
| Pinecone Serverless/Pod | Dynamic: switches between pre/post based on selectivity | Unpredictable latency under varying selectivity |
| Weaviate | Integrated into HNSW traversal | Predictable |
| Qdrant | Payload filter runs during search | Predictable |
| pgvector | SQL WHERE clause, same query plan | Predictable; requires index on filtered columns |

Pinecone's dynamic switching is the most common source of unexplained p99 spikes in production. When filter selectivity changes — for example, a tenant with few documents versus one with millions — query latency varies unpredictably and you cannot override the behavior.

## 3. Pricing model breakdown

### Pinecone Serverless

Pinecone Serverless bills in **read units** and **write units**.

One read unit equals one call to the `/query` endpoint. Cost is per call, not per vector scanned or result returned. A query that scans 10M vectors costs the same as one that scans 10K.

One write unit equals one call to `/upsert`. You can batch multiple vectors in a single call.

| Volume | Monthly read unit cost (approx.) |
|---|---|
| 1M queries | ~$0.20 |
| 100M queries | ~$20 |
| 1B queries | ~$200 |
| 10B queries | ~$2,000 |

Storage costs approximately $0.84 per GB-month. A 100M-vector index at 768 dimensions runs roughly 100 GB, adding ~$84/month.

The free tier includes 1M read units and 125K write units per month.

The pricing model penalizes high query volume with predictable linearity. At 1B queries/month, Serverless costs more than a pod-based configuration. The break-even point is somewhere between 100M and 1B queries/month depending on index size.

### Pinecone Pod-based

Pod pricing is fixed per hour. The rate stays constant regardless of utilization.

| Pod type | Approximate monthly cost | Intended workload |
|---|---|---|
| s1 | ~$96 | Small indexes, low QPS |
| p1 | ~$576 | Standard production |
| p2 | ~$2,304 | High throughput, large indexes |

A single p1 pod with a 100 GB index costs approximately $601/month. [FACT-CHECK: Research notes indicate pod storage pricing is $0.25/GB/month for metadata + index, making 100 GB storage cost ~$25 and bringing the total to ~$601/month. The article omits this storage cost breakdown that the notes reference.] Throughput scales linearly with pod count. There is no per-query charge, so high-volume use cases favor pods over Serverless above a certain query threshold.

The pod model punishes small or variable workloads. The minimum meaningful production configuration (one p1 pod) costs $576/month before storage. Idle time costs the same as peak time.

### Weaviate Cloud

Weaviate Cloud bases its pricing on **dimensions stored** — the product of vector count and dimensionality. Storing 10M vectors at 1536 dimensions (OpenAI text-embedding-3-large) uses 15.36 billion dimensions. Pricing tiers are based on this figure plus compute allocation.

Weaviate does not publish a simple per-unit price publicly. The platform sizes tiers for specific dimension ranges with associated compute. The practical implication: switching to a higher-dimensional embedding model increases your Weaviate Cloud bill directly, independent of query volume. Lower-dimensional embeddings (384-dim models) cost meaningfully less to store.

A sandbox tier is available for development. Production tiers require contacting sales for exact quotes at scale. [FACT-CHECK: Missing — the research notes mention a "free tier" for Weaviate (described as a "sandbox tier"), but they also note WCS launched in 2023. The article's description of requiring sales contact for production tiers is not explicitly supported or contradicted in the provided research notes, which appear to be cut off mid-section]

### Qdrant Cloud

Qdrant Cloud prices clusters by **RAM allocation**. Vectors must fit in RAM for HNSW to operate at advertised latency. Scalar quantization reduces the effective RAM requirement, making it the practical tool for cost control.

Cluster pricing scales with RAM and replica count. A single-node cluster with 8 GB RAM runs roughly $50–100/month depending on region. High-availability (multi-node) configurations multiply that cost. Qdrant Cloud publishes a pricing calculator; sizing requires knowing your vector count, dimensionality, and whether quantization is applied.

The RAM-based model is transparent: you pay for what the index occupies. Quantization directly reduces cost.

### pgvector

pgvector adds no cost of its own. You pay for Postgres compute and storage.

| Host | Free tier | Production cost model |
|---|---|---|
| Neon | Yes (auto-suspend after 5 min idle) | Compute-hours + storage |
| Supabase | Yes (paused after 1 week inactive) | Compute + storage tier |
| AWS RDS | No | Instance-hour + storage |
| Self-hosted | N/A | Infrastructure only |

Neon's serverless model auto-suspends compute after 5 minutes of inactivity. The next connection takes 1–3 seconds to resume. On Neon's paid plans, compute can be set to always-on. Supabase offers connection pooling via PgBouncer, which mitigates some cold-start effects from serverless functions.

For workloads already running on Postgres, adding pgvector costs nothing in licensing and often costs nothing in infrastructure if the existing instance has headroom.

## 4. Operational model

### Index health, replication, and backups

| Database | Replication | Backup | Index health monitoring |
|---|---|---|---|
| Pinecone (all tiers) | Managed | Managed | Managed |
| Weaviate Cloud | Managed | Managed | Managed |
| Qdrant Cloud | Managed; HA clusters available | Snapshots (managed) | Managed |
| pgvector (hosted) | Host-dependent (RDS: automated; Neon: automated) | Host-dependent | Host-dependent |
| Weaviate self-hosted | Your responsibility | Your responsibility | Your responsibility |
| Qdrant self-hosted | Your responsibility | Native snapshot API | Your responsibility |

Self-hosted Weaviate and Qdrant transfer all operational responsibility to you. Qdrant's native snapshot API simplifies backup scripting. Weaviate requires backup configuration via its backup module.

### Horizontal scaling

Pinecone Serverless scales automatically. Pinecone Pod-based requires you to add pods manually; there is no auto-scaling. Weaviate Cloud scales within your tier automatically. Qdrant Cloud requires manual cluster resizing but offers sharding configuration. pgvector scales with the underlying Postgres instance — vertical scaling is the primary lever, with read replicas available on most hosts for read-heavy workloads.

Qdrant self-hosted uses manual sharding. You define the shard count at collection creation. Resharding without downtime is not supported in all versions.

### Multi-tenancy patterns

Three common patterns for isolating tenant data:

1. **Namespace/collection per tenant**: Each tenant gets its own index or collection. Provides strong isolation. Increases management overhead at hundreds of tenants.
2. **Metadata filtering**: All tenants share an index; a `tenant_id` field filters results per query. Lower overhead but no storage isolation.
3. **Separate instances**: Maximum isolation, maximum cost.

| Database | Native multi-tenancy support |
|---|---|
| Pinecone | Namespaces within an index; or separate indexes |
| Weaviate | Multi-tenancy per Collection (explicit API support) |
| Qdrant | Payload-based filtering; or separate collections |
| pgvector | Row-level security + SQL WHERE clause |

Weaviate has explicit multi-tenancy built into its Collection API, with tenant-level enable/disable controls. This is an architectural feature, not a convention. For SaaS RAG applications with many tenants, this distinction matters for both isolation guarantees and operational tooling.

### Cold start behavior

Pinecone Serverless can experience 1–5 second delays after an idle period. The idle threshold is not documented precisely. This affects the first query after inactivity but not sustained traffic. Weaviate Cloud and Qdrant Cloud do not exhibit cold starts under normal operation. Neon's auto-suspend causes 1–3 second delays on the first connection after the idle timeout. Pinecone Pod-based and RDS-hosted pgvector have no cold start behavior.

For Vercel deployments, Serverless functions themselves cold-start independently of the vector database. Compounding cold starts — Vercel function plus Pinecone Serverless idle, or Vercel function plus Neon auto-suspend — can produce multi-second first-query latency.

## 5. Integration with the Vercel stack

### Edge-compatible clients vs. Node.js-only SDKs

Vercel Edge Runtime restricts available APIs: no Node.js native modules, limited `fs`, no `net` module. Any SDK that uses Node.js TCP connections directly will not run at the Edge.

| Database | Edge Runtime compatible | Notes |
|---|---|---|
| Pinecone | Yes | `@pinecone-database/pinecone` uses fetch; Edge-compatible |
| Weaviate | Partial | Official JS client uses Node.js HTTP; not Edge-compatible without workaround |
| Qdrant | Partial | `@qdrant/js-client-rest` uses fetch; Edge-compatible. gRPC client is not. |
| pgvector | No | All Postgres clients (`pg`, `postgres.js`) use TCP; not Edge-compatible |

For pgvector on Edge, Neon provides a serverless driver (`@neondatabase/serverless`) that tunnels queries over HTTP/WebSocket and is Edge-compatible. Supabase provides a similar HTTP-based Data API.

### Connection pooling for pgvector from serverless functions

Standard Postgres drivers open a TCP connection per client. Serverless functions may spawn hundreds of concurrent instances, each opening its own connection. Postgres has a hard connection limit (typically 100–500 depending on instance size). Without pooling, you will hit this limit under moderate concurrency.

Solutions:
- **Neon**: Built-in connection pooling via their serverless proxy. Set `?pooling=true` in the connection string or use the serverless driver.
- **Supabase**: PgBouncer available in transaction mode. Use the pooler connection string, not the direct connection string.
- **RDS**: Deploy PgBouncer separately, or use RDS Proxy (adds cost and latency).

Connection exhaustion is the most common operational failure when moving pgvector from development to production on serverless infrastructure.

### TypeScript SDK maturity

| Database | TypeScript SDK | Maturity notes |
|---|---|---|
| Pinecone | `@pinecone-database/pinecone` | First-party, well-maintained, typed |
| Weaviate | `weaviate-ts-client` | First-party, typed, but lags behind REST API features |
| Qdrant | `@qdrant/js-client-rest` | First-party, typed, REST-based |
| pgvector | `pgvector` npm package | Thin wrapper; most logic stays in SQL |

### Vercel AI SDK integration

The Vercel AI SDK (`ai` package) includes a retrieval abstraction. As of early 2024, it does not include first-party vector store adapters for any of these databases. Integration requires writing a retrieval function that queries your chosen database and returns documents in the format the AI SDK expects. The SDK is database-agnostic at the retrieval layer; any HTTP-accessible vector store works with a thin adapter.

## 6. When to reach for each

### Pinecone Serverless

**Best fit**: Variable or unpredictable query volume; teams that want zero infrastructure management; projects under 10M vectors; early-stage products where operational simplicity outweighs cost optimization.

**Worst fit**: High and sustained query volumes above ~500M/month where per-query costs exceed pod costs; applications requiring predictable p99 latency when filter selectivity varies; workloads sensitive to cold-start latency.

### Weaviate

**Best fit**: Applications where embedding generation and storage need to be colocated; multi-tenant SaaS with many tenants (explicit multi-tenancy API); teams comfortable with GraphQL; use cases requiring built-in vectorizer pipeline management.

**Worst fit**: Edge Runtime deployments (JS client incompatibility); teams that want a simple REST API without a GraphQL layer; cost-sensitive workloads where Weaviate Cloud dimension-based pricing scales faster than expected.

### Qdrant

**Best fit**: High-throughput production workloads; applications with complex metadata filtering requirements; teams self-hosting and needing predictable performance from a Rust-based engine; cost-sensitive deployments using scalar quantization to reduce RAM requirements.

**Worst fit**: Teams that need a fully managed, hands-off experience and prefer not to manage cluster sizing; applications tightly integrated with the Vercel Edge Runtime that need gRPC performance.

### pgvector

**Best fit**: Applications already on Postgres where vectors are one feature among many. This suits workloads that combine vector search with relational queries (joins, aggregations, transactions); teams that want one fewer infrastructure component; projects where Neon or Supabase is already in use.

**Worst fit**: Datasets above ~10M vectors where query latency becomes a concern; workloads requiring very high query throughput (>5,000 QPS) without significant Postgres hardware investment; teams that need sophisticated ANN index tuning without SQL expertise.

The "use pgvector" argument holds when your data model is inherently relational and vector search is not the performance-critical path. It breaks down when you cross into tens of millions of vectors or when p99 latency under concurrent load is a hard requirement.

### The two most common mistakes

**Choosing Pinecone Serverless for a high-volume, stable workload.** Serverless pricing targets unpredictable traffic patterns. A steady 500M queries/month costs ~$100. [FACT-CHECK: Based on the research notes' pricing of ~$0.20 per million read units, 500M queries/month would cost ~$100 — this figure is consistent. However, the notes flag the pricing figures as needing verification against current Pinecone docs] The same workload on a single p1 pod costs $576 but never cold-starts and provides better p99. Engineers prototype on Serverless, see it work, and then discover the cost cliff at scale when they should have moved to pods — or to Qdrant — at the first capacity review.

**Deploying pgvector without a connection pooler on serverless infrastructure.** The development experience on a local Postgres instance or a single Neon database is smooth. Moving to production on Vercel with dozens of concurrent serverless function instances exhausts Postgres connection limits within minutes of real traffic. The fix is mechanical — add PgBouncer or use Neon's pooling connection string — but it is not apparent from the pgvector documentation and causes real outages in teams that skip it.
